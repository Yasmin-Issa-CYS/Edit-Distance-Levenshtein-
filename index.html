📖 Algorithm Overview ▾

The Levenshtein Distance (or Edit Distance) between two strings is the minimum number of single-character operations — insertions, deletions, or substitutions — needed to transform one string into the other. Introduced by Vladimir Levenshtein in 1965.

Recurrence: d[i][j] = min( d[i-1][j]+1, d[i][j-1]+1, d[i-1][j-1]+cost ) where cost=0 if s[i]=t[j], else 1.
Complexity: O(mn) time, O(mn) space. Optimizable to O(min(m,n)) space using two rolling rows.
Applications: Spell checking, DNA sequence alignment, natural language processing, file diff tools, plagiarism detection.

Comparison with Alternative Algorithms

Levenshtein DP ✓

Builds a full (m+1)×(n+1) table bottom-up. Uses optimal substructure to guarantee correctness.

Time: O(mn) Correct ✓

Naïve Recursion

Recursively tries all 3 operations at each step. Recomputes overlapping subproblems many times.

Time: O(3^max(m,n)) Very Slow ✗

Memoized Recursion

Top-down DP with memoization. Same asymptotic complexity as bottom-up but with function-call overhead.

Time: O(mn) Correct ✓

Ukkonen's Algorithm

Optimized for small edit distances (k). Runs in O(kn) time — much faster when k≪n.

Time: O(kn) Faster for small k
Pseudocode with Complexity Annotations
function EditDistance(s[1..m], t[1..n]):
  // O(mn) space — full table
  for i=0 to m: d[i][0]=i   // O(m)
  for j=0 to n: d[0][j]=j   // O(n)
  for i=1 to m:                 // outer loop: O(m)
    for j=1 to n:               // inner loop: O(n) → total O(mn)
      cost = (s[i]≠t[j]) ? 1 : 0 // O(1)
      d[i][j] = min(d[i-1][j]+1, d[i][j-1]+1, d[i-1][j-1]+cost) // O(1)
  return d[m][n]                 // Total: O(mn) time, O(mn) space
CLO-2 — Searching & the Traceback Path

In Levenshtein DP, searching plays a critical role in two ways: (1) character matching — each cell checks if s[i]=t[j], analogous to a character search; (2) the traceback — after the table is filled, we search for the optimal path from d[m][n] back to d[0][0] by greedily following minimum-cost predecessors.

Traceback as Search: Starting at d[m][n], at each step we look at 3 neighbors (↑, ←, ↖) and move to the one that gave the minimum cost. This greedy search through the already-computed DP table finds the optimal edit sequence in O(m+n) time.

Character Matching: The Hidden Search

At every cell d[i][j], the algorithm compares s[i] with t[j]. This is a direct character-equality search:

if s[i] = t[j]:  ← character search / match
  cost = 0   ← free diagonal move
else:
  cost = 1   ← search fails → substitution cost
What Happens Without Search (Matching)

With Character Matching

cost=0 when s[i]=t[j]. The diagonal move is free, reducing the final edit distance. Matching is essentially a "search for equality."

Edit dist "kitten"→"sitting" = 3

Without Matching (always cost=1)

If we ignore character equality and always use cost=1, every diagonal move costs 1 too. The algorithm degenerates and always returns max(m,n).

Would give dist = max(m,n) always
Traceback Algorithm
function Traceback(d, s, t, m, n):
  i=m, j=n, path=[]
  while i>0 or j>0:               // O(m+n) search
    match = (s[i]=t[j])
    if d[i][j] = d[i-1][j-1] + (match?0:1): // diagonal?
      i--; j--; path.add(match ? MATCH : SUBSTITUTE)
    elif d[i][j] = d[i][j-1] + 1:           // insert?
      j--; path.add(INSERT)
    else: i--; path.add(DELETE)        // delete
  return reverse(path)
CLO-3 — Dynamic Programming Paradigm
Paradigm: Dynamic Programming (Bottom-Up)
Levenshtein distance is a classic DP problem. We solve it by building solutions to smaller subproblems first, then combining them — never recomputing the same subproblem twice.
Four DP Properties Present in Levenshtein
PropertyPresent?Evidence
Optimal Substructure✓ Yesd[i][j] = min(d[i-1][j]+1, d[i][j-1]+1, d[i-1][j-1]+cost)
Overlapping Subproblems✓ YesNaïve recursion recomputes d[i][j] exponentially many times
Memoization / Tabulation✓ YesBottom-up table stores every d[i][j] exactly once
Polynomial Complexity✓ YesO(mn) vs O(3^max(m,n)) naïve — exponential speedup
Overlapping Subproblems — Why Naïve Recursion Fails

Consider the naïve recursive approach. To compute EditDistance("ki", "si"), we must compute EditDistance("k","s"), EditDistance("k","si"), and EditDistance("ki","s"). Each of those in turn computes smaller subproblems, many of them identical:

ED(2,2)
 ├─ ED(1,2) ─┬─ ED(0,2), ED(1,1)★, ED(0,1)
 ├─ ED(2,1) ─┬─ ED(1,1)★, ED(2,0), ED(1,0)
 └─ ED(1,1)★ ← computed 3× without memoization!
★ Overlapping Subproblem: ED(1,1) is needed by multiple parent calls. Without DP, it's recomputed exponentially. DP computes it once and stores the result in the table.
DP Table Filling Order (Bottom-Up Tabulation)

The table is filled in strict row-major order. Cell d[i][j] only ever depends on cells that have already been computed: d[i-1][j] (above), d[i][j-1] (left), and d[i-1][j-1] (diagonal). This guarantees correctness.

Counterexample: Why Greedy Fails

A greedy approach might try to make the locally cheapest operation at each character position. But greedy is not optimal for edit distance:

Greedy Attempt: "ab" → "ba"

Greedy sees a≠b, substitutes a→b: "bb". Then b≠a, substitutes b→a: "ba". Result: 2 ops.

2 operations (suboptimal)

DP Optimal: "ab" → "ba"

DP finds: delete 'a' then insert 'a' at end (or just 2 substitutions). Also 2 ops — but DP guarantees it's minimal.

2 operations (guaranteed optimal)
Why Greedy Can't Work: A greedy choice now may force expensive operations later. DP avoids this by considering ALL subproblem solutions before committing.

🔑 Key Concepts ▾
Optimal Substructure: The optimal solution to EditDistance(s[0..i], t[0..j]) contains optimal solutions to smaller subproblems. This is WHY dynamic programming works here.
Three Operations:
• Insert: add t[j] to s → use d[i][j-1] + 1
• Delete: remove s[i] → use d[i-1][j] + 1
• Replace: change s[i] to t[j] → use d[i-1][j-1] + 1
• Match: s[i]=t[j], no op needed → use d[i-1][j-1] + 0
Common Mistakes:
• Forgetting that d[i][j] depends on THREE predecessors, not one
• Confusing rows/columns (s goes down rows, t goes across columns)
• Thinking a match is always better than a replacement — sometimes an insert+delete is cheaper
• Using 1-indexed vs 0-indexed strings incorrectly in the recurrence
🔬 Space Optimization: O(min(m,n)) ▾

The full DP table uses O(mn) space. But notice that d[i][j] only depends on the current row and the previous row! We can keep just two rows and alternate:

prev = [0,1,2,...,n]   // base case row
for i=1 to m:
  curr = [i, 0, 0, ...]  // first col base case
  for j=1 to n:
    cost = (s[i]≠t[j]) ? 1 : 0
    curr[j] = min(prev[j]+1, curr[j-1]+1, prev[j-1]+cost)
  prev = curr
return prev[n]         // O(n) space only!
Trade-off: Space drops from O(mn) to O(n), but we lose the ability to reconstruct the traceback (edit sequence). To get the edit sequence, we need the full table or a divide-and-conquer approach (Hirschberg's algorithm, O(mn) time, O(m+n) space).
